Baruch Underground Listing of Professor Ability

GitHub Repository

A school-specific professor review site written in PHP. Not affiliated with the City University of New York or Baruch College.

The Problem: During my short time at Baruch College, many students relied on word of mouth when picking and choosing classes. Understandably, students wanted the best professor for their classes. Although sites like RateMyProfessor exist, the reviews for Baruch College professors were either outdated or nonexistent - it simply could not keep up with the influx of new adjunct professors that taught at Baruch College, a prevalent problem throughout the CUNY system.

The Solution: A Baruch College specific professor review website. The inspiration to create a school specific professor review website came from CULPA, a site that's allowed Columbia students to review Columbia professors/classes since 1997.

Challenges

  1. Reducing/eliminating spam submissions
    • Minimum length in submission box [see code]
    • Capturing users IP address [see code]
      • Because I was collecting a review submitters IP address, I disclosed this in the Privacy Policy
    • All reviews are assigned a '0' value in 'isApproved' [see code]
      • This meant all user submitted reviews were manually reviewed. To make this easier on myself, I created a moderator page that queried the database to list all reviews with the value '0' (false) in 'isApproved' [see code]
    • Implementing Google reCaptcha
  2. Searching for Reviews & Courses
    • As all data was cleansed before being published, I used the 'LIKE' MySQL operator to query reviews [see code]
      • In hindsight, I recognize that this was not an efficient way to search reviews - an influx of users may very well have crashed the site
  3. Timestamping Reviews
    • To make the process of submitting a review more streamlined, I opted to internally fetch the timestamp of the newly created row instead of asking what the day and time is from the user
      • When querying the database for the timestamp, I would simply convert the unix timestamp using the date() function [see code]
  4. Requiring the same code in every page
    • I wanted every page to look the same and follow the same design principles - but if I wanted to change one thing, I needed to change it in every single file.
      • Although rudimentary, I decided to use the include statement throughout my website [see code]
        • In hindsight, using a template engine like Twig may have been a better idea
  5. Sharing Specific Reviews (Permalinks)
    • Because every single review had a unique identifier, I could query a specific review
      • e.g. /review.php?id=5 would fetch the review with 'id' 5

Database Schema

  • reviews
    • id - Unique identifier for each review
    • fname - First name of professor
    • lname - Last name of professor
    • course - The course code (e.g. BPL 5100)
    • department - Department the course falls under
    • review - User submitted review
    • timestamp - Timestamp when the review was submitted
    • isApproved - Boolean data type
    • submit_ip - Stores IP address of the review submitter

Statistics

  • 942 Users Visited (85% from NY)
  • 9,511 Pages Viewed
  • 200+ Professors & Courses Anonymously Reviewed
  • Most users prefered to search and read reviews rather than submit reviews (as expected)
